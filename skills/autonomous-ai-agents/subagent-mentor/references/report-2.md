# Руководство по настройке мультиагентной CLI-среды: делегирование, отслеживание и управление субагентами в рамках одной сессии

## Краткое резюме (TL;DR)

Современные CLI AI-агенты (Claude Code, OpenCode, Hermes, Antigravity CLI, Cursor Agent CLI) поддерживают создание специализированных субагентов — изолированных рабочих единиц с собственным контекстом, инструментами и системными инструкциями, которые выполняют задачи параллельно или последовательно под управлением основной сессии. Ключевой принцип: все субагенты остаются в рамках одной сессии, их задачи отслеживаются через встроенные механизмы (TodoWrite/TodoRead, Agent View, `/agents` overlay), а прогресс мониторится в реальном времени. Для каждого инструмента существует свой формат конфигурации субагентов (Markdown + YAML frontmatter для Claude Code, JSON/JSONC для OpenCode, YAML для Hermes, динамическое определение для Antigravity), но архитектурный паттерн един: оркестратор делегирует задачи специалистам, получает результаты и синтезирует итоговый ответ. Настройка мультиагентной среды требует: (1) определения ролей субагентов с четкими описаниями и ограничениями инструментов, (2) настройки механизмов отслеживания задач, (3) конфигурации мониторинга прогресса, (4) обеспечения изоляции контекста и безопасности. Ниже приведены пошаговые инструкции для каждого инструмента и единый шаблон инструкции, который можно передать AI CLI агенту для обучения.

## Часть 1. Общая архитектура мультиагентной CLI-среды

### 1.1. Паттерн «оркестратор — исполнители»

Любая мультиагентная система в CLI строится на паттерне «оркестратор — исполнители» (supervisor-worker). Основная сессия выступает в роли оркестратора: она анализирует запрос пользователя, разбивает его на подзадачи, определяет необходимые специализации, создает субагентов с соответствующими ролями и делегирует им работу. Каждый субагент работает в изолированном контексте — у него свой системный промпт, свой набор доступных инструментов, свое ограничение по токенам. После завершения работы субагент возвращает результат оркестратору, который синтезирует итоговый ответ. Этот паттерн реализован во всех рассматриваемых инструментах, хотя и с различными техническими деталями.

В Claude Code оркестратор использует инструмент **Agent** (ранее Task) для создания субагентов, передавая им имя типа и промпт. В OpenCode механизм аналогичен — через инструмент **Task** с указанием `subagent_type`. Hermes использует инструмент **delegate_task** с параметрами `goal` и `context`. Antigravity CLI работает через команду **`/goal`**, где оркестратор автоматически определяет необходимые роли субагентов. Cursor Agent CLI имеет более ограниченные возможности мультиагентности — в основном через последовательное выполнение задач в рамках одной сессии с использованием **`--continue`** и **`--resume`**.

Важнейшее свойство этого паттерна — **изоляция контекста**. Субагент не видит историю разговора основной сессии, не имеет доступа к предыдущим вызовам инструментов оркестратора и начинает работу с «чистого листа». Это предотвращает «загрязнение контекста» — накопление нерелевантной информации в основной сессии, которое приводит к деградации качества ответов и перерасходу токенов. Оркестратор передает субагенту только необходимый минимум контекста через параметры вызова.

### 1.2. Механизмы отслеживания задач в рамках одной сессии

Для отслеживания задач в рамках одной сессии используются три взаимодополняющих механизма. Первый — **встроенные инструменты TodoWrite/TodoRead**, доступные в Claude Code и OpenCode. Это глобальное состояние задач per session: агент создает список задач с помощью TodoWrite, обновляет статус (pending, in_progress, completed, cancelled), читает текущий список через TodoRead. В OpenCode oh-my-opencode этот механизм дополнен хуком todo-continuation-enforcer, который автоматически напоминает агенту о незавершенных задачах.

Второй механизм — **Agent View / Dashboard**. Claude Code предоставляет **`claude agents`** — полноэкранный интерфейс, показывающий все фоновые сессии со статусами (Working, Needs input, Idle, Completed, Failed). Hermes имеет **`/agents`** (alias `/tasks`) overlay — дерево субагентов с kill/pause controls, стоимостью и токенами per branch. OpenCode имеет **opencode-monitor** — отдельный дашборд для отслеживания сессий. Эти инструменты дают визуальный контроль над выполнением задач без необходимости переключения между терминалами.

Третий механизм — **Shared Task List с dependency graph**. Claude Code Agent Teams используют разделяемый список задач в `~/.claude/tasks/{team-name}/` с поддержкой зависимостей blockedBy: задача автоматически разблокируется, когда завершены ее зависимости. Это позволяет строить сложные пайплайлы с условным выполнением, сохраняя все в рамках одной логической сессии.

### 1.3. Принципы изоляции и безопасности

Изоляция субагентов — это не только архитектурное удобство, но и мера безопасности. Каждый субагент получает строго ограниченный набор инструментов (principle of least privilege): code reviewer получает только Read, Grep, Glob — он не может модифицировать файлы; implementer получает Read, Write, Edit, Bash — но не может использовать WebFetch. Это предотвращает нежелательные побочные эффекты и снижает поверхность атаки.

Межагентный trust boundary требует, чтобы каждый субагент имел независимые полномочия, не наследуя полный доступ оркестратора. Делегирование не должно незаметно эскалировать привилегии: если оркестратор может создавать субагентов, сами субагенты не должны автоматически получать те же права. Все действия субагентов логируются, что позволяет проводить аудит и обнаруживать аномалии.

## Часть 2. Claude Code — нативные субагенты и Agent Teams

### 2.1. Создание субагентов через Markdown + YAML frontmatter

В Claude Code субагенты определяются как Markdown-файлы с YAML frontmatter. Это текстовые файлы, которые легко версионировать, делить с командой и редактировать вручную или программно. Файл состоит из двух частей: метаданных в формате YAML между разделителями `---` и системного промпта в формате Markdown ниже.

#### 2.1.1. Размещение файлов: `.claude/agents/` vs `~/.claude/agents/`

Существует два уровня размещения субагентов. **Проектные субагенты** размещаются в директории `.claude/agents/` в корне проекта. Они специфичны для конкретного кодовой базы, наследуют контекст проекта и могут быть закоммичены в git для совместного использования командой. Приоритет у проектных субагентов выше — если имя конфликтует с пользовательским субагентом, выбирается проектный.

**Пользовательские субагенты** размещаются в `~/.claude/agents/` и доступны во всех проектах. Это персональные помощники для типовых задач — code review, debugging, написание документации. Поддиректории поддерживаются (например, `~/.claude/agents/review/security.md`), но путь не влияет на идентификатор — только поле `name` в frontmatter.

Приоритет разрешения конфликтов: проектные > пользовательские > плагиновые. Если два файла в одной области объявляют одинаковое имя, Claude Code сохраняет один и отбрасывает другой без предупреждения — поэтому важно поддерживать уникальность имен.

#### 2.1.2. Полная спецификация frontmatter: name, description, tools, model, memory, background, effort, isolation, color, initialPrompt

Полная спецификация frontmatter для Claude Code субагента включает следующие поля:

Поле **name** (обязательное) — уникальный идентификатор субагента. Используйте kebab-case (нижний регистр с дефисами), например: `code-reviewer`, `security-auditor`, `api-developer`. Это имя используется для явного вызова через `@name`.

Поле **description** (обязательное, самое важное) — натуральное языковое описание, когда и зачем вызывать этого субагента. Именно по этому полю Claude Code принимает решение об автоматическом делегировании. Хорошее описание включает: триггеры (какие запросы активируют субагента), ожидаемый результат, формат вывода. Примеры: `Proactively reviews code for quality, security, and maintainability. Use when reviewing PRs or checking code before commit.` или `Debugging specialist for errors, test failures, and unexpected behavior. MUST BE USED when encountering any errors or test failures.` — ключевые слова MUST BE USED и use proactively повышают вероятность автоматического вызова.

Поле **tools** (необязательное, но критически важное для безопасности) — список разрешенных инструментов через запятую. Если опущено, субагент наследует все инструменты основной сессии. Рекомендуемые наборы: для read-only анализа — `Read, Grep, Glob`; для анализа с командной строкой — `Read, Grep, Glob, Bash`; для написания кода — `Read, Write, Edit, Bash, Grep, Glob`; для исследования — `Read, Grep, Glob, WebFetch, WebSearch`.

Поле **model** (необязательное) — модель для субагента. Допустимые значения: `haiku` (быстрый и дешевый, для поиска и анализа), `sonnet` (сбалансированный, для реализации), `opus` (максимальное качество, для сложных архитектурных решений), `inherit` (наследовать модель основной сессии). Оптимизация затрат: запускайте основную сессию на Opus, а субагентов для рутинных задач на Haiku.

Поле **memory** (необязательное, новое) — персистентная память между сессиями. Значения: `user` (память в `~/.claude/agent-memory/<agent-name>/`, персональная) или `project` (память в `.claude/agent-memory/<agent-name>/`, общая для команды). При запуске первые 200 строк из MEMORY.md внедряются в системный промпт. Это позволяет субагенту накапливать знания о кодовой базе, паттернах, решениях.

Поле **permissionMode** (необязательное) — режим разрешений: `default` (подтверждение для опасных действий), `acceptEdits` (авто-одобрение правок файлов), `dontAsk` (чтение без подтверждений), `bypassPermissions` (полный пропуск, использовать осторожно), `plan` (только планирование).

Поле **background** (необязательное) — если `true`, субагент всегда выполняется как фоновая задача. Поле **effort** — уровень усилий: `low`, `medium`, `high`, `xhigh`, `max`. Поле **isolation** — `worktree` для выполнения в изолированной git worktree. Поле **color** — цвет для визуальной идентификации в UI. Поле **initialPrompt** — автоматически отправляемый промпт при старте.

#### 2.1.3. Пример: субагент для code review с ограниченными правами

```markdown
---
name: code-reviewer
description: Proactively reviews code for quality, security, and maintainability. 
  Use when reviewing PRs, checking code before commit, or analyzing changed files.
  MUST BE USED for any code review task.
tools: Read, Grep, Glob, Bash
model: sonnet
permissionMode: dontAsk
memory: project
---

## Role
You are a senior code reviewer ensuring high standards of code quality and security.

## When Invoked
1. Run `git diff` to see recent changes
2. Focus on modified files only
3. Begin review immediately without asking clarifying questions

## Review Checklist
- Code is clear and readable
- No exposed secrets or API keys
- Proper error handling
- Good test coverage
- No SQL injection or XSS vulnerabilities
- Performance considerations addressed

## Output Format
Provide feedback organized by priority:
- Critical issues (must fix) — marked with [CRITICAL]
- Warnings (should fix) — marked with [WARNING]  
- Suggestions (consider improving) — marked with [SUGGESTION]

For each issue include: file path, line number, problem description, recommended fix.
```

Этот субагент имеет read-only доступ к файлам плюс Bash для запуска git diff, использует сбалансированную модель Sonnet, не запрашивает подтверждений (только чтение), и накапливает проектную память о паттернах кодовой базы.

### 2.2. Интерактивное создание через `/agents` UI

Помимо ручного создания файлов, Claude Code предоставляет интерактивный интерфейс для управления субагентами. Команда **`/agents`** открывает полноэкранный UI со списком всех доступных субагентов (встроенных, пользовательских, проектных, плагиновых). В этом интерфейсе можно: просматривать существующих субагентов, создавать новых через guided setup или с помощью Claude (Generate with Claude), редактировать конфигурацию и доступ к инструментам, удалять кастомных субагентов.

Процесс создания через UI состоит из шагов: (1) открыть `/agents`, (2) переключиться на вкладку Library, (3) выбрать Create new agent → Personal (для `~/.claude/agents/`) или Project (для `.claude/agents/`), (4) выбрать Generate with Claude и описать желаемого субагента на естественном языке, (5) Claude сгенерирует имя, описание и системный промпт, (6) выбрать набор инструментов (Read-only tools для reviewer, все инструменты для разработчика), (7) выбрать модель (Haiku для быстрых задач, Sonnet для баланса, Opus для сложных), (8) выбрать цвет для визуальной идентификации, (9) настроить память (User scope для персональных знаний, Project scope для командных, None если не нужно), (10) сохранить — субагент сразу доступен без перезапуска.

Важно: на январь 2026 существует баг — файловые субагенты в `~/.claude/agents/` иногда не обнаруживаются автоматически. Обходное решение: создавать субагентов через `/agents` UI, они сохраняются в профиль и доступны в любой сессии. Альтернативно, можно перезагрузить сессию после добавления файлов.

### 2.3. Делегирование: автоматическое и явное

Claude Code поддерживает два режима делегирования. **Автоматическое делегирование** работает на основе поля description субагента. Когда пользователь отправляет запрос, Claude Code анализирует его, сопоставляет с описаниями доступных субагентов и автоматически выбирает подходящего. Например, запрос «проверь этот PR» сопоставляется с description code-reviewer и активирует его. Автоматическая делегация удобна, но не 100% надежна — для критичных workflow рекомендуется явное делегирование.

**Явное делегирование** осуществляется через синтаксис **`@agent-name`** в промпте. Например: `@code-reviewer проверь изменения в src/auth/` или `@debugger разберись с этой ошибкой в логах`. Это гарантирует вызов конкретного субагента независимо от автоматического анализа. Явное делегирование особенно важно: когда нужно контролировать какой именно субагент выполняет задачу, при параллельном запуске нескольких специалистов, в скриптах и автоматизациях.

### 2.4. Мониторинг в рамках одной сессии

#### 2.4.1. TodoWrite/TodoRead — отслеживание задач

В рамках одной сессии Claude Code использует встроенные инструменты **TodoWrite** и **TodoRead** для отслеживания задач. TodoWrite создает или заменяет список задач с статусами: `pending`, `in_progress`, `completed`, `cancelled`. TodoRead возвращает текущий список. Это позволяет агенту вести структурированный план работы и отслеживать прогресс.

Однако есть ограничение: обновления TodoWrite внутри Task tool не видны пользователю в реальном времени — отображается только финальный результат. Для прозрачности рекомендуется инструктировать субагентов возвращать структурированный отчет о проделанной работе вместо сырых выводов.

#### 2.4.2. `claude agents` — Agent View для мониторинга всех сессий

**Agent View** — это полноэкранный интерфейс, открываемый командой `claude agents` (или клавишей `\` внутри Claude Code). Он показывает все фоновые сессии в едином списке с группировкой по статусам: **Working** (анимированная иконка — агент активно работает), **Needs input** (желтая иконка — агент ждет решения пользователя), **Idle** (приглушенная — сессия готова к работе), **Completed** (зеленая — задача завершена), **Failed** (красная — ошибка), **Stopped** (серая — остановлена Ctrl+X).

Каждая строка показывает: идентификатор сессии, текущую активность, время последнего изменения, иконку статуса процесса (живая сессия или завершенный процесс). Можно открыть **peek panel** (Space) — панель быстрого просмотра, показывающую что нужно от сессии и последний вывод. Можно отправить reply прямо из peek panel без необходимости attach. Enter открывает полную сессию для прямого взаимодействия.

Фоновые сессии не требуют открытого терминала — supervisor процесс поддерживает их работу даже после закрытия shell. Состояние сохраняется на диске и переживает перезагрузки. Для фильтрации используются: `a:<name>` — сессии с конкретным агентом, `s:<state>` — по статусу, `#<number>` — по PR.

#### 2.4.3. Task list с dependency graph для Agent Teams

Для сложных мультиагентных workflow Claude Code предоставляет **Agent Teams** — архитектуру с разделяемым списком задач. Task list хранится в `~/.claude/tasks/{team-name}/` в формате JSON. Каждая задача имеет: статус (pending, in_progress, completed), владельца (какой агент выполняет), зависимости blockedBy. Когда задача-зависимость завершается, зависимые задачи автоматически разблокируются.

Для использования shared task list между сессиями установите переменную окружения: `CLAUDE_CODE_TASK_LIST_ID="my-project" claude`. Все сессии с одинаковым ID разделяют один список задач с real-time обновлениями. Это позволяет запускать несколько Claude Code сессий, работающих над одним проектом, с координацией через единый task list.

#### 2.4.4. Встроенный `/cost` и сторонние дашборды (claude-view)

Для отслеживания затрат внутри сессии используйте команду **`/cost`** — она показывает накопленные токены и оценочную стоимость. Для расширенного мониторинга существуют сторонние инструменты: **claude-view** (Rust, `npx claude-view`) — дашборд с real-time отслеживанием всех сессий, токенов, костов, sub-agent trees, полнотекстовым поиском и 85 MCP tools; **Claude Code Agent Monitor** (Node.js/React) — полноценный дашборд с Kanban board, аналитикой, WebSocket обновлениями и VS Code extension.

### 2.5. Практический пример: pipeline из 3 субагентов

Пример pipeline для разработки фичи: (1) `pm-spec` анализирует требование и пишет спецификацию, (2) `architect-review` валидирует дизайн и создает ADR, (3) `implementer-tester` реализует код и тесты. В Claude Code это выглядит как:

```
User: Добавить OAuth2 PKCE для auth модуля
→ Claude автоматически делегирует @pm-spec
  → pm-spec читает codebase, пишет спецификацию, сохраняет в docs/specs/oauth2-pkce.md
→ Когда spec готов, Claude делегирует @architect-review  
  → architect-review анализирует спецификацию, создает ADR в docs/decisions/
→ Когда ADR готов, Claude делегирует @implementer-tester
  → implementerTester реализует код, пишет тесты, запускает pytest
→ Claude синтезирует результат: код реализован, тесты проходят, PR готов
```

Каждый субагент работает в изоляции, возвращает структурированный результат, основная сессия сохраняет чистый контекст.

## Часть 3. OpenCode — Task tool и oh-my-opencode

### 3.1. Конфигурация в `opencode.json` / `opencode.jsonc`

OpenCode использует JSON/JSONC формат конфигурации. Файл может размещаться на трех уровнях: глобальный (`~/.config/opencode/opencode.json`), проектный (корень проекта), путь через переменную `OPENCODE_CONFIG`. Приоритет: переменная > проект > глобальный.

#### 3.1.1. Определение primary агентов

Primary агенты — это основные агенты, которые вы вызываете напрямую через промпт при старте OpenCode. Пример конфигурации:

```json
{
  "$schema": "https://opencode.ai/config.json",
  "agent": {
    "build": {
      "mode": "primary",
      "prompt": "You are a build expert. Focus on compiling, testing, and optimizing code.",
      "tools": {
        "read": true, "write": true, "edit": true, "bash": true,
        "grep": true, "glob": true, "todoread": true, "todowrite": true
      }
    },
    "plan": {
      "mode": "primary", 
      "prompt": "{file:./prompts/plan.txt}",
      "tools": {
        "write": false, "edit": false, "bash": false
      }
    }
  }
}
```

Поле `mode: primary` обозначает, что это основной агент. Поле `prompt` может содержать текст или ссылку на файл (`{file:./prompts/plan.txt}`). Поле `tools` определяет доступные инструменты — `true` для разрешения, `false` для запрета.

#### 3.1.2. Определение subagent-ов с mode: subagent

Subagent-ы определяются с `mode: subagent` и полем `description`, по которому основной агент принимает решение о делегировании:

```json
{
  "$schema": "https://opencode.ai/config.json",
  "agent": {
    "code-reviewer": {
      "description": "Reviews code for best practices and potential issues",
      "mode": "subagent",
      "prompt": "You are a code reviewer. Focus on security, performance, and maintainability.",
      "tools": {
        "write": false,
        "edit": false
      }
    },
    "security-auditor": {
      "description": "Security-focused code auditor. Use for auth, crypto, input validation review.",
      "mode": "subagent",
      "prompt": "You are a security auditor. Check for OWASP Top 10, injection flaws, insecure dependencies.",
      "tools": {
        "read": true, "grep": true, "glob": true,
        "write": false, "edit": false, "bash": false
      }
    }
  }
}
```

Ключевое отличие от Claude Code: в OpenCode субагенты определяются в едином JSON файле, а не как отдельные Markdown файлы. Однако oh-my-opencode плагин добавляет поддержку Markdown-определений аналогичных Claude Code.

### 3.2. Делегирование через Task tool

В OpenCode делегирование происходит через инструмент **Task** (также доступен как `@agent-name`). Механизм: основной агент вызывает Task с параметром `subagent_type`, указывая тип субагента, и `prompt` с задачей. Task tool создает новую сессию для субагента, предоставляет нужные инструменты и системный промпт, запускает выполнение.

Важные особенности: субагент stateless — нельзя отправлять дополнительные сообщения, субагент не может общаться с внешним миром кроме финального отчета; промпт должен быть максимально детальным — включите все необходимый контекст; можно запускать несколько агентов параллельно — отправьте несколько вызовов Task в одном сообщении; субагент всегда доверяет выводам — укажите явно, если нужна критическая оценка.

### 3.3. oh-my-opencode — расширенная мультиагентность

oh-my-opencode (OMO) — популярный плагин для OpenCode, добавляющий продвинутую мультиагентную систему с 11+ специализированными агентами.

#### 3.3.1. Специализированные агенты: Sisyphus, Hephaestus, Oracle и др.

OMO предоставляет иерархию агентов, каждый оптимизирован под конкретную модель и задачу: **Sisyphus** (Opus/Kimi K2.6) — главный оркестратор, классифицирует намерения, делегирует задачи; **Hephaestus** (GPT-5.3+) — автономный deep worker для end-to-end задач; **Oracle** (GPT-5.2) — read-only архитектурный консультант и отладчик; **Prometheus** (Opus/Kimi) — стратегический планировщик; **Atlas** (Sonnet/Kimi) — orchestrator todo-листов; **Explore** (Haiku/Grok) — быстрый поиск по codebase; **Librarian** (GLM) — поиск документации и GitHub; **Metis** — pre-planning консультант; **Momus** — ревьюер планов.

Делегирование происходит через `delegate_task(category=...)` для category-based маршрутизации или `call_omo_agent()` для прямого вызова по имени. Оба поддерживают параллельное фоновое выполнение.

#### 3.3.2. Team Mode с tmux-визуализацией

Team Mode (v4.0, opt-in) превращает OMO из «один агент с субагентами» в полноценную мультиагентную систему. Lead агент оркестрирует команду из до 8 параллельных членов, коммуницирующих через dedicated tools (`team_create`, `team_send_message`, `team_task_create`, `team_status`). Визуализация происходит в tmux layout с focus + grid windows — можно наблюдать как каждый член команды работает одновременно.

Для включения: установите `team_mode.enabled: true` в `.opencode/oh-my-openagent.jsonc`, перезапустите OpenCode. Команды размещаются в `~/.omo/teams/{name}/config.json` (user) или `<project>/.omo/teams/{name}/config.json` (project).

### 3.4. Отслеживание: todowrite/todoread и opencode-monitor

OpenCode имеет встроенные инструменты **todowrite** и **todoread**, работающие аналогично Claude Code. oh-my-opencode расширяет это системой todo-continuation-enforcer, который автоматически напоминает агенту о незавершенных задачах. Однако будьте осторожны — при недоступности todowrite возможен infinite loop (обход: добавьте `"todo-continuation-enforcer"` в `disabled_hooks`).

Для визуального мониторинга используйте **opencode-monitor** — дашборд с: единой панелью для всех сессий, desktop-нотификациями, instant access к attach (`t`) или browser (`b`), иерархическим деревом parent/child субагентов, отслеживанием токенов и cost в real-time, color-coded предупреждениями о контексте.

### 3.5. Session management: attach, resume, parent-child relationships

OpenCode поддерживает управление сессиями: `opencode attach` — подключение к running сессии, `opencode ls` — список сессий, resume через `--resume`. oh-my-opencode добавляет session management с конфигурируемым количеством запоминаемых child сессий (`sessionManager.maxSessionsPerAgent`, default 2). Это позволяет возобновлять работу субагентов, сохраняя контекст прочитанных файлов.

## Часть 4. Hermes Agent — delegate_task и orchestrator role

### 4.1. Конфигурация делегирования в `~/.hermes/config.yaml`

Hermes Agent использует YAML формат конфигурации. Секция **delegation** контролирует поведение субагентов:

```yaml
delegation:
  model: "google/gemini-3-flash-preview"     # Модель для субагентов (пусто = наследовать родителя)
  provider: "openrouter"                      # Провайдер (пусто = наследовать)
  base_url: "http://localhost:1234/v1"        # Прямой endpoint (приоритет над provider)
  api_key: "local-key"                        # API key для base_url
  max_concurrent_children: 3                  # Параллельных субагентов (мин 1, без максимума)
  max_spawn_depth: 1                          # Глубина дерева (1-3, 1 = плоское)
  orchestrator_enabled: true                  # Глобальный kill switch для orchestrator role
  max_iterations: 50                          # Максимум ходов на субагента
  child_timeout_seconds: 600                  # Таймаут в секундах (10 мин по умолчанию)
```

### 4.2. delegate_task: goal, context, toolsets, role

Инструмент **delegate_task** — основной механизм делегирования в Hermes. Параметры: `goal` — что субагент должен достичь; `context` — файлы, ошибки, структура проекта, ограничения (всё необходимое для работы); `toolsets` — набор доступных инструментов; `role` — `orchestrator` для субагентов, которые могут далее делегировать; `max_iterations` — ограничение ходов для конкретного вызова.

Примеры toolset patterns: `["terminal", "file"]` — для кода и отладки; `["web"]` — для исследования; `["terminal", "file", "web"]` — full-stack (default); `["file"]` — read-only анализ. Некоторые toolsets блокируются для leaf subagents: `delegation` (только orchestrator может делегировать), `clarify` (субагенты не взаимодействуют с пользователем), `memory` (нет записи в shared memory), `send_message` (нет cross-platform side effects).

### 4.3. Batch mode: параллельное выполнение до 3 субагентов

Hermes поддерживает batch mode — передача массива задач для параллельного выполнения:

```python
delegate_task(tasks=[
    {"goal": "Research competitor pricing for Tool A", "toolsets": ["web"]},
    {"goal": "Research competitor pricing for Tool B", "toolsets": ["web"]},
    {"goal": "Fix lint errors in src/utils.ts", "toolsets": ["terminal", "file"]}
])
```

По умолчанию запускается до 3 параллельных субагентов. Результаты возвращаются в порядке индексов задач, независимо от порядка завершения. Если задач больше лимита — возвращается ошибка (не silent truncation). Настройте лимит через `max_concurrent_children` или переменную `DELEGATION_MAX_CONCURRENT_CHILDREN`.

Batch mode идеален для: исследовательских задач (три web-only субагента исследуют разные темы одновременно), code tasks когда каждый субагент работает с разным файлом или модулем. Избегайте для последовательно зависимых задач — субагенты не видят что делают другие.

### 4.4. Orchestrator role и многоуровневая иерархия (max_spawn_depth)

По умолчанию каждый субагент — leaf node, он не может делегировать дальше. Для многоуровневых иерархий используйте `role="orchestrator"`:

```python
delegate_task(
    goal="Survey three deployment approaches and recommend one",
    role="orchestrator",
    context="We need to deploy a Python API. Consider Docker, serverless, and VM options."
)
```

Orchestrator child сохраняет инструмент `delegation` и может создавать leaf grandchildren. Глубина контролируется `max_spawn_depth`: 1 (default) — плоская структура, 2 — orchestrator children могут создавать leaf grandchildren, 3 — трехуровневое дерево. Стоимость растет мультипликативно: при max depth 3 и concurrency 3 теоретически до 27 параллельных leaf агентов. На практике двухуровневые иерархии (depth 2) покрывают большинство workflow.

### 4.5. Мониторинг через `/agents` overlay

Hermes предоставляет **`/agents`** overlay (alias `/tasks`) в TUI для real-time мониторинга. Возможности: **Live tree view** — дерево запущенных и завершенных субагентов, сгруппированных по родителю; **Per-branch cost and token rollups** — какая подзадача потребляет больше всего ресурсов; **File-touch summaries** — какие файлы читал/модифицировал каждый субагент; **Kill and pause controls** — отмена конкретного субагента mid-flight без влияния на siblings; **Post-hoc review** — пошаговая история каждого субагента после завершения.

Полезный паттерн: запустить batch delegation, наблюдать в overlay за отстающими, убить child который занимает более чем вдвое больше времени чем siblings, и перезапустить эту подзадачу с более явным контекстом.

### 4.6. Профили для изоляции контекста между проектами

Hermes поддерживает **profiles** — независимые конфигурации с изолированными сессиями, skills, memory. Полезно для разделения work/personal проектов или разных команд. Каждый профиль имеет свой `config.yaml`, сессии, skills, cron jobs. Переключение между профилями обеспечивает полную изоляцию контекста.

## Часть 5. Antigravity CLI (agy) — динамические субагенты

### 5.1. Установка и базовая настройка

Antigravity CLI (`agy`) — преемник Gemini CLI, написанный на Go. Установка:

```bash
# macOS / Linux
curl -fsSL https://antigravity.google/cli/install.sh | bash

# Windows PowerShell
irm https://antigravity.google/cli/install.ps1 | iex
```

После установки запустите `agy` и аутентифицируйтесь с Google аккаунтом. CLI поддерживает 8 моделей: Gemini 3.x Flash (Low/Medium/High), Gemini 3.x Pro, Claude Sonnet 4.6, Claude Opus 4.6, GPT-OSS 120B — все на одной Google AI Pro/Ultra подписке.

### 5.2. Команда `/goal` — автоматическое определение ролей субагентов

Ключевая особенность Antigravity — **динамические субагенты**. Вместо предварительной конфигурации ролей, оркестратор сам определяет необходимые специализации на основе запроса. Команда **`/goal`** запускает полностью автономное выполнение без промежуточных подтверждений:

```
/goal I have a sales dataset at ./sales_raw.csv.
Spin up multiple subagents to clean, analyze, and visualize it.
Produce a final HTML report and save all outputs in this folder.
```

Процесс: Stage 1 — Environment profiling (проверка Python, библиотек, прав доступа); Stage 2 — Planning and subagent definition (создание implementation_plan.md, определение data_cleaner, data_analyzer, data_visualizer); Stage 3 — Sequential execution (data_cleaner сначала); Stage 4 — Parallel execution (analyzer и visualizer одновременно после очистки); Stage 5 — Assembly (сборка финального отчета).

Субагенты определяются через **DefineSubagent** — вы не создаете конфигурационные файлы, оркестратор делает это автоматически на основе профилирования задачи.

### 5.3. Хуки в формате JSON для lifecycle events

Antigravity поддерживает **JSON hooks** — перехват и кастомизация поведения агента на lifecycle stages: before tool call, after model call, at loop stop conditions. Это позволяет интегрировать внешние системы, логировать действия, добавлять валидацию. Формат hooks определяется в JSON конфигурации.

### 5.4. Плагины и миграция с Gemini CLI

Плагины (бывшие extensions) добавляют capabilities: `agy plugin import gemini` — миграция с Gemini CLI; поддержка skills, MCP servers, hooks. На первом запуске предлагается автоматическая конвертация extensions в plugins.

### 5.5. Безопасность: human-in-the-loop подтверждения

Даже с `/goal` CLI запрашивает one-time permission перед классами операций (bash commands). Это deliberate design choice — вы одобряете класс операций один раз, и далее запросы не повторяются. Это сохраняет контроль без замедления автоматизации.

## Часть 6. Cursor Agent CLI — session management и resume

### 6.1. Установка и команды: `cursor-agent`, `cursor-agent ls`, `cursor-agent resume`

Cursor Agent CLI — инструмент для запуска Cursor Agent из терминала. Установка: `curl https://cursor.com/install -fsSL | bash`. Основные команды: `cursor-agent` — интерактивная сессия; `cursor-agent -p "prompt"` — non-interactive режим; `cursor-agent ls` — список чатов; `cursor-agent resume` — возобновление последнего чата; `cursor-agent --resume <chatId>` — возобновление конкретного чата; `cursor-agent --continue` — продолжение предыдущей сессии.

### 6.2. Режимы: agent (default), ask (read-only), plan

`--mode=agent` (default) — полноценный агент с возможностью модификации файлов. `--mode=ask` — read-only режим для анализа без изменений. `--mode=plan` — генерация плана перед реализацией.

### 6.3. Session persistence: `--continue`, `--resume <chatId>`

Сессии Cursor CLI chat-based. Для возобновления: получите chat ID из `cursor-agent ls` или web UI, передайте его через `--resume <chatId>` или `--continue` для последней сессии. Каждая возобновленная сессия продолжает тот же conversation context.

Важное ограничение: Cursor CLI не предоставляет нативного мультиагентного делегирования как Claude Code или Hermes. Параллельная работа достигается через запуск нескольких независимых сессий в разных терминалах, возможно с git worktrees для изоляции.

### 6.4. Ограничения мультиагентности и обходные пути

Cursor CLI имеет ограниченные возможности мультиагентности по сравнению с Claude Code. Рекомендуемые обходные пути: использование **git worktrees** для параллельной работы над разными ветками; **CCManager** для управления несколькими Cursor сессиями; интеграция с **Entire** для трекинга сессий и checkpoint management; **HAPI** web app для мониторинга сессий и approve permissions с телефона.

## Часть 7. Единая инструкция для AI CLI агента — шаблон

Этот раздел содержит готовый шаблон инструкции, который вы можете передать своему AI CLI агенту. Агент прочитает его и научит вас настраивать мультиагентную среду для конкретного инструмента.

### 7.1. Шаблон инструкции (скопируйте и отправьте агенту)

```
# ИНСТРУКЦИЯ: Настройка мультиагентной CLI-среды

## Твоя роль
Ты — эксперт по настройке мультиагентных CLI AI-сред. Твоя задача — научить 
пользователя настраивать субагентов, делегировать им задачи и отслеживать 
прогресс, оставаясь в рамках одной сессии.

## Общие принципы (применимы ко всем инструментам)
1. Один субагент = одна ответственность. Не смешивай роли.
2. Описание (description) — самое важное поле. По нему принимается решение 
   об автоматическом делегировании. Будь конкретным, включай триггеры и 
   ожидаемый результат.
3. Принцип минимальных прав: давай субагенту только необходимые инструменты.
   Read-only агент не нуждается в Write.
4. Общие правила проекта в CLAUDE.md / AGENTS.md — не дублируй их в каждом 
   субагенте.
5. Все субагенты работают в рамках одной сессии — используй TodoWrite/TodoRead 
   для отслеживания, Agent View / /agents overlay для мониторинга.
6. Модель оптимизации: Haiku для поиска/анализа, Sonnet для реализации, 
   Opus для сложных архитектурных решений.

4. Настрой мониторинг (Agent View, /agents overlay, или opencode-monitor)

6. Оптимизируй: анализируй cost, корректируй модели и concurrency

→ Проверь автоматическое делегирование

```

### 7.2. Как адаптировать шаблон под конкретный инструмент

Для адаптации шаблона под конкретный инструмент замените «Поинструментальные руководства» на соответствующую секцию. Если используете несколько инструментов — оставьте все секции и добавьте контекст о том, какой инструмент primary. Для интеграции между инструментами (например, Hermes делегирует в Claude Code) используйте **hermes-agent-acp-skill** или аналогичные bridge-решения.

### 7.3. Примеры промптов для обучения через агента

Примеры промптов, которые можно отправить агенту после передачи шаблона: «Покажи как создать субагента для code review в Claude Code с ограниченными правами» → агент создаст файл `.claude/agents/code-reviewer.md` с правильным frontmatter. «Настрой в Hermes delegate_task для параллельного анализа 3 файлов» → агент покажет batch mode конфигурацию. «Объясни как работает /goal в Antigravity на примере» → агент продемонстрирует процесс с реальным выводом.

## Часть 8. Best practices и безопасность

### 8.1. Принцип «один субагент = одна ответственность»

Самый важный принцип проектирования субагентов — единственная ответственность. Субагент, который делает более одной различной работы, снова становится монолитом. Если ему нужен длинный системный промпт, покрывающий несколько доменов — разделите его. Это упрощает отладку, повышает переиспользуемость и снижает вероятность неожиданного поведения.

### 8.2. Оптимизация затрат: модель Haiku для поиска, Sonnet для реализации, Opus для архитектуры

Оптимизация затрат достигается правильным выбором моделей. Используйте **Haiku** (или эквивалент — Gemini Flash, GPT-4o-mini) для: поиска по codebase, grep-like операций, чтения документации, простого анализа. Используйте **Sonnet** (или эквивалент) для: написания кода, рефакторинга, отладки, code review. Используйте **Opus** (или эквивалент) только для: архитектурных решений, сложного планирования, security audit, критичных дизайн-решений.

Паттерн маршрутизации: запускайте основную сессию на сильной модели (Opus), но делегируйте рутинные задачи на более дешевые модели. Например, анализ 100 файлов через Explore на Haiku стоит в 10-20 раз дешевле, чем на Opus.

### 8.3. Изоляция через git worktrees для параллельной работы

Для параллельной работы нескольких агентов над одним репозиторием используйте **git worktrees**. Это позволяет иметь несколько рабочих директорий, связанных с одним `.git`, но с разными ветками:

```bash
git worktree add ../myproject-feature-auth -b feature/auth
git worktree add ../myproject-feature-api -b feature/api
cd ../myproject-feature-auth && claude "implement OAuth2"
cd ../myproject-feature-api && claude "add rate limiting"
```

Каждый агент работает в изоляции, конфликтов при записи не происходит. После завершения — review и merge каждой ветки отдельно.

### 8.4. Минимальные права доступа (principle of least privilege)

Явно ограничивайте инструменты для каждого субагента. Code reviewer не нуждается в Write. Researcher не нуждается в Bash. Не полагайтесь на системный промпт для предотвращения использования инструментов — модель может быть prompt-инжектирована из инструмента, которого у нее просто нет. Enforce scope на уровне allowed-tools — модель не может вызвать инструмент, который ей не предоставлен.

### 8.5. Обработка ошибок и таймаутов субагентов

Субагенты могут застревать в циклах, потреблять токены без прогресса или таймаутиться. Механизмы защиты: **Kill controls** — Hermes и Claude Code позволяют убить конкретного субагента без влияния на siblings; **max_iterations** — ограничение ходов на субагента (default 50 в Hermes); **child_timeout_seconds** — wall-clock таймаут (10 мин default в Hermes); **Post-hoc review** — после завершения просмотрите пошаговую историю для понимания что пошло не так.

Паттерн восстановления: если субагент фейлится, запустите его с более явным контекстом или разбейте задачу на мелкие части. Для критичных workflow используйте retry с exponential backoff.

### 10.2. Когда использовать мультиагентность, а когда — один агент

Используйте **одного агента** когда: задача простая и линейная (< 3 связанных шагов), быстрый фикс в одном файле, нет естественных точек параллелизма, контекст укладывается в окно модели. Накладные расходы на оркестрацию не окупаются для тривиальных задач.

Используйте **мультиагентность** когда: проект растягивается на несколько сессий, работа имеет естественные зависимости, несколько субагентов будут сотрудничать, нужна оптимизация контекстного окна (изоляция тяжелых задач), критична персистентность прогресса. Практическое правило: если задача требует > 3 связанных шагов или может выиграть от параллельного выполнения — используйте Tasks/Subagents.
